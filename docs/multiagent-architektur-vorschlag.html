ALLRIS · Architekturvorschlag

Von der Workflow-Sammlung zum Redaktionssystem

Sechs Agenten mit je einer Aufgabe, ein zentrales Redaktionsgedächtnis und ein Orchestrator, der die Freigabe-Logik an einer Stelle hält statt in zehn Nodes verteilt. Kein neuer Workflow pro Bug — ein System, das aus sich selbst lernt.

Basis: P3 Bewertung, P3b Repair/Guard, P4 Content-Reaktion, P5 Visual Prompt Builder Status: Konzept zur Diskussion, noch nicht umgesetzt

Der Unterschied, den das macht

Heute laufen sechs unabhängig getriggerte Workflows, die alle selbst auf status/visualStatus lesen und schreiben. Jeder neue Bug wird an der Fundstelle gepatcht — meist als weitere Zeile in einem ohnehin riesigen Prompt. Das Ergebnis: eine AI Analyse JSON mit ~44.000 Zeichen, eine AI Visual JSON mit ~39.000 Zeichen, SourceLock-Prüfungen an fünf verschiedenen Stellen mit je eigener Logik.

Heute

6 Workflows, die sich gegenseitig über Statuswerte "zurufen". SourceLock wird von der gleichen KI-Instanz erzeugt, die auch Relevanz bewertet und Satire briefet. Jede neue Regel wächst in denselben Mammut-Prompt.

Vorschlag

1 Orchestrator, 6 reine Funktionsagenten ohne eigenen Datenbank-Zugriff, 1 QA-Vertrag für alle Stufen, 1 Lern-Speicher statt Prompt-Wachstum. Fakten und Bewertung sind strukturell getrennt, nicht nur thematisch.

Die Redaktionsstrecke

Jeder Agent bekommt sauberes JSON, liefert sauberes JSON zurück — keiner schreibt selbst in die Data Table. Das erledigt allein der Orchestrator, direkt nach jedem QA-Durchlauf. Damit verschwindet die "Node ersetzt Item"-Falle, die heute dreimal unabhängig voneinander Bugs verursacht hat.

Die sechs Agenten im Detail

Jede Karte zeigt die Aufgabe, den Vertrag (Input/Output) und — genauso wichtig — was der Agent ausdrücklich nicht tun darf. Diese Grenze ist QA-prüfbar, nicht nur eine Prompt-Anweisung.

01 · Sachbearbeitung

Fakten-Agent

Extrahiert ausschließlich belegbare Informationen aus der Ratsvorlage. Jedes Feld muss auf eine Textstelle zurückführbar sein.

InputcombinedText, title, Vorgangskette
Output (= neuer SourceLock) 
{ sourceTopic, sourceConflict,
  requiredTerms[], requiredObjects[],
  requiredAction, affectedGroups[],
  concreteNumbers[], deadlines[],
  responsibilities[], documentType,
  quotesVerbatim[] }

Ausgeschlossen: Bewertung, Tonalität, Pointe, jede Formulierung, die nicht wörtlich im Dokument steht.

Ersetzt: den extrahierenden Teil von "AI Zusammenfassung" + die SourceLock-Erzeugung, die heute in "AI Analyse JSON" mitläuft.

02 · Sachbearbeitung

Analyse-Agent

Bewertet politische Relevanz, Bürgerwirkung, Konflikte und Ungleichbehandlung — ausschließlich auf Basis der Fakten, ohne eigenen Dokumentzugriff.

Inputfacts (read-only, kein Rohtext)
Output 
{ politicalRelevance, buergerwirkung,
  empfehlung, conflictType,
  affectedGroupsRisk,
  kommunikationspotenzial, reasoning }

Ausgeschlossen: Satire, Bildmotive, neue Fakten — jede Aussage in reasoning muss aus facts.* zitierbar sein.

Ersetzt: den Bewertungsteil von "AI Analyse JSON", ohne satireBrief/satireAngle/visualAnchors.

03 · Redaktion

Satire-Agent

Entwickelt mehrere Satire-Ansätze mit unterschiedlicher Schärfe — verankert in den Fakten, nicht in der vorgegebenen Tonlage der Analyse.

Inputfacts (read-only) + empfehlung/conflictType nur als Schärfe-Richtwert
Output 
{ variants: [
  { sharpness: mild|mittel|scharf,
    angle, punchlines[],
    targetInstitution,
    headline, subline,
    requiredTermUsed[] }, ... ] }

Ausgeschlossen: targetInstitution darf nie eine betroffene/schutzbedürftige Gruppe sein; jede Pointe braucht ≥1 Bezug zu requiredTerms/-Objects.

Ersetzt: satireBrief/-Angle/-Punchlines aus "AI Analyse JSON" + "AI Headline Varianten" (P5) — heute an zwei Stellen, künftig an einer.

04 · Redaktion

Bild-Agent

Entwickelt nur Motividee und Bild-Prompt zu genau einer gewählten Satire-Variante — keine eigene Schlagzeile, keine eigene Schärfe-Entscheidung.

Input1 gewählte Variante + facts.requiredObjects/sourceTopic
Output 
{ motifIdea, imagePrompt,
  negativePrompt, mustShowObjects[],
  styleNotes }

Ausgeschlossen: Headline/Subline-Text (kommt fertig vom Satire-Agenten), Bewertung der Schärfe.

Ersetzt: den Prompt-Bauteil von "AI Visual JSON" (P5) — heute zusätzlich mit Headline-Echo, Style-Lock und Sensible-Gruppen-Check in einem Prompt gebündelt.

05 · Kontrolle

QA-Agent

Sucht aktiv nach Fehlern, Halluzinationen und SourceLock-Verstößen. Bestätigt nichts von sich aus — liefert entweder Verstöße oder eine leere Liste.

Input{ stage, producedOutput, facts, priorArtifacts }
Output 
{ passed: bool,
  violations: [{rule, evidence, severity}],
  hallucinationSuspects[],
  sourceLockBreaches[] }

Ausgeschlossen: Kein "sieht gut aus"-Freigabepfad; ein Agent, ein Vertrag für alle vier Content-Stufen.

Ersetzt: P3-Validierung, P3b Pass 1+2, P4s zwei unabhängige SourceLock-Checks, P5s canonical-overwrite-Logik — heute fünf verstreute, teils inkonsistente Prüfungen.

06 · Gedächtnis

Lern-Agent

Speichert akzeptierte und abgelehnte Beispiele und macht sie für künftige Läufe abrufbar — kein eigener Content, keine Bewertung.

Zwei Funktionen 
record(stage, output, qaVerdict, humanVerdict)
retrieve(stage, context) -> top 3-5 Beispiele

Ausgeschlossen: Erzeugt selbst keine Inhalte; wirkt nur über Beispiele, die andere Agenten in ihren Prompt ziehen.

Verhindert das Muster "neue Regel = neue Zeile im Mammut-Prompt" (z.B. die wachsende Liste sensibler Metaphern in P3b).

Eine State Machine statt zwanzig Statuswerten

Heute schreiben über zehn Nodes in vier Workflows unabhängig voneinander status/visualStatus (siehe Architektur-Review: über 20 verschiedene Werte im Umlauf). Im neuen System schreibt nur der Orchestrator den Fortschritt, direkt nach einem QA-Ergebnis — nirgendwo sonst.

pipelineStageBedeutungNächster Schritt
fakten_pendingWartet auf Fakten-Agent→ QA → analyse_pending
analyse_pendingFakten geprüft, wartet auf Bewertung→ QA → satire_pending oder ignoriert/beobachtet (Endzustand)
satire_pendingWartet auf Satire-Varianten→ QA → awaiting_headline_choice
awaiting_headline_choiceMensch wählt Variante (Matrix)→ bild_pending
bild_pendingWartet auf Motiv + Prompt→ QA → published
qa_retry(n)QA hat abgelehnt, Wiederholung mit QA- und Lern-Kontextzurück an denselben Agenten, max. n Versuche
qa_escalatedRetry-Limit erreichtMatrix-Ops-Alarm (bestehendes Muster bleibt)
publishedEndzustand, Content live—

Migrationsweg

Kein Big-Bang-Umbau. Jede Phase ist für sich lauffähig und liefert bereits Wert, bevor die nächste beginnt.

  1. Geringstes Risiko

    QA-Agent konsolidieren

    Die fünf verstreuten Prüfungen (P3, P3b ×2, P4 ×2, P5) in einen Vertrag zusammenführen — noch ohne die anderen Agenten anzufassen.

  2. Kann sofort parallel starten

    Lern-Speicher aufsetzen

    Neue Data Table allris_lernbeispiele; ab Tag 1 jede QA-Entscheidung und jede Matrix-Auswahl/-Reroll protokollieren, auch bevor ein Agent sie nutzt.

  3. Fakten/Analyse trennen

    Die 44K-Zeichen "AI Analyse JSON" aufteilen: Extraktion raus in den Fakten-Agenten, Bewertung bleibt, Zugriff auf Rohtext für die Bewertung kappen.

  4. Satire-Agent bilden

    satireBrief-Teil aus der Analyse plus "AI Headline Varianten" (P5) zu einem Agenten mit mehreren Schärfegraden zusammenführen.

  5. Bild-Agent verschlanken

    "AI Visual JSON" auf Motiv+Prompt reduzieren; Headline-Echo, Style-Lock-Text und Guard-Logik wandern in Satire-Agent bzw. QA-Agent.

  6. Größter struktureller Schnitt, zuletzt

    Orchestrator ablösen

    Die sechs unabhängig getriggerten Workflows durch einen zentralen State-Machine-Owner ersetzen — erst wenn die Agentengrenzen sich in Phase 1–5 bewährt haben.

Was sich dadurch konkret ändert

  • Qualität: QA prüft aktiv gegen die Fakten des Fakten-Agenten, nicht gegen sich selbst — Halluzinationen fallen strukturell auf, nicht zufällig.
  • Wartbarkeit: Sechs kleine, einzeln testbare Verträge statt vier Mammut-Prompts mit vermischten Zuständigkeiten.
  • Weniger Prompt-Wildwuchs: Neue Regeln werden Lern-Beispiele, keine weiteren Zeilen in einem 44K-Zeichen-Text.
  • Eine Wahrheit für den Zustand: Ein pipelineStage, geschrieben von genau einer Stelle, statt zwanzig verstreuter Statuswerte.
